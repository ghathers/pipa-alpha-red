

# 🏗️ Google AI Studio Assignment (Oxford Edition)
**Scenario:** Post-Incident Planning App (PIPA) – Project Alpha Red

## Overview
This assignment guides you through building a production-ready Post-Incident Planning App (PIPA) using Google AI Studio, the latest Gemini models, Stitch, and optional Cloud Run deployment. You will ingest highly technical forensic data (Project Alpha Red Final Incident Report) and executive summaries (Post-Incident Summary Slides), synthesizing them into actionable, regulatory-compliant post-incident meeting insights that map to a highly structured, card-based interface.

## 🧠 Phase 2: The Required Workflow

### Step 1: Create a specialized Gem to generate structured meeting insights.
In Google AI Studio, create a new Prompt. You will act as the "Gem" architect by defining the core intelligence.
* **Action:** Paste the following into the System Instructions block:
    > *"You are the intelligence engine for a Post-Incident Planning App (PIPA). Your objective is to ingest technical forensic cyber reports and executive presentation slides, analyze them for truth-conflicts, and generate structured data to populate a technical post-incident dashboard."*

### Step 2: Define output schema for summary, risks, talking points, and next steps.
To ensure the output perfectly maps to your card-based UI, you must constrain the model to a strict JSON schema. 
* **Action:** Add this exact JSON schema to your System Instructions:

```json
{
  "incident_kpis": {
    "systems_impacted": "String or 'TBD'",
    "accounts_compromised": "String or 'TBD'",
    "target_remediation_timeframe": "String",
    "exfiltrated_data": "String (e.g., 2.2TB PII)"
  },
  "executive_summary": {
    "narrative": "Detailed paragraph explaining the attack vector and impact.",
    "incident_dates": "String",
    "threat_actor": "String"
  },
  "root_cause_analysis": [
    {
      "stage": "Initial Access | Credential Access | Lateral Movement",
      "description": "String detailing the technical compromise."
    }
  ],
  "recovery_timeline_tasks": [
    {
      "phase": "Containment | Regulatory | Remediation",
      "task_description": "String",
      "estimated_timescale": "String",
      "priority": "Integer",
      "owner": "String"
    }
  ],
  "timeline_key": {
    "C": "Containment",
    "R": "Regulatory",
    "Rem": "Remediation"
  },
  "briefing_tab_data": {
    "talking_points": ["Executive script points"],
    "regulatory_risks": ["Identified compliance gaps"]
  }
}
```

### Step 3: Generate UX screens using Stitch from structured descriptions.
Before iterating your prompt in AI Studio, generate your baseline UI.
* **Action:** Take a sample JSON output generated by your schema and paste it into **Google Stitch**.
* **Prompt Stitch:** *"Create a highly structured, vertically stacked Post-Incident Planning App (PIPA) dashboard in a minimalist light-gray theme with stark black cards and neon green accents. Include top KPI blocks, an Executive Summary card with sub-boxes for 'Incident Dates' and 'Threat Actor', a numbered Post-Incident Plan list, a 'Root Cause Analysis' section broken down by attack phase, and a 'Recovery Timeline & Tasks' table. Add a bottom navigation bar with icons for Dashboard, Risk, Briefing, and Timeline."*

### Step 4: Export screens as PNG files.
* **Action:** Once Stitch renders the dashboard components, export the generated screens to your local drive as `.png` files. Focus specifically on capturing a screenshot of the dashboard showing the remediation timeline.

### Step 5: Import screens into Google AI Studio.
* **Action:** Return to your prompt in Google AI Studio. Use the **"+" (Insert)** button to upload your Stitch `.png` files alongside your incident report and slides. This prepares the model for a multimodal reasoning task.

### Step 6: Iterate and refine prompts with version history tracking.
* **Iteration 1 (Baseline):** Generate the JSON from the text files using the Gemini 3.1 Flash Preview model. Save to Version History.
* **Iteration 2 (Truth-Checking):** Add a rule: *"Cross-reference the incident context. Ensure the 'incident_dates' and 'exfiltrated_data' KPI fields reflect the forensic truth."* Save to Version History.
* **Iteration 3 (Multimodal Refinement):** Add a rule: *"Analyze the uploaded screenshot of the dashboard showing the remediation timeline. Improve the timeline with estimated timescales, priority order, and a key sumarising the card labels (C, R, R etc.)."* Save to Version History.

### Step 7: Demonstrate prompt diffs and reasoning improvements.
* **Action:** Open the **Version History** panel. Take screenshots showing the evolution from Iteration 1 to Iteration 3. Ensure your documentation logs how the prompt diffs forced the model to enrich the data specifically for the UI's constraints.

### Step 8: Sync project to GitHub repository.
Create a new GitHub repository (e.g., `pipa-alpha-red`) and upload:
1.  `assignment.md`
2.  `reflection.md`
3.  Your exported Stitch `.png` files.
4.  Your AI Studio Version History screenshots.

### Step 9: Optionally deploy application to Cloud Run and test endpoint.
* **Action:** In AI Studio, click **"Get Code"** (Python). Wrap the SDK code in a lightweight Flask application and use `gcloud run deploy` to push it to a live endpoint. Include the URL in your GitHub repo.

## 📂 Appendix A: `assignment.md`
*Here are my notes for assignment.md*

> # Architecture and Design Decisions: Post-Incident Planning App (PIPA)
> 
> ## 1. Architectural Overview
> The Post-Incident Planning App (PIPA) is designed to ingest highly technical forensic data and executive summaries, synthesizing them into a rigid, data-dense dashboard. The application utilizes a multimodal Retrieval-Augmented Generation (RAG) pipeline powered by the **Gemini 3.1 Flash Preview** model via Google AI Studio.
> 
> ## 2. Core Design Decisions
> * **Model Selection:** The **Gemini 3.1 Flash Preview** was selected for its ability to execute advanced reasoning with near-instant latency. It successfully handled the "Conflict Resolution Logic" required to cross-reference the optimistic Executive Slides against the stark reality of the Forensic Report.
> * **Structured Output (JSON):** The AI's output was strictly constrained to a predefined JSON schema mapped explicitly to the visual hierarchy of the UI, ensuring discrete buckets for KPIs, Root Cause Analysis, and the Recovery Timeline.
> * **Multimodal Grounding:** By importing the Stitch PNGs back into AI Studio during the final iteration, the model gained structural awareness of the UI. This allowed it to analyze the exact visual layout of the remediation timeline table and programmatically generate missing contextual data (like a label key) required by the design.
> 
> ## 3. Prompt Diffs & Reasoning Improvements
> * **V1 to V2:** Added truth-checking constraints. The model successfully cross-referenced the incident context, overriding the executive narrative to ensure that the `incident_dates` and `exfiltrated_data` KPIs reflected the factual forensic truth.
> * **V2 to V3:** Added the multimodal UI constraint. The model analyzed the uploaded screenshot of the dashboard showing the remediation timeline. It successfully enriched the output by inferring and appending estimated timescales, sorting the JSON array into a logical priority order, and generating a `timeline_key` object to explain the shorthand labels (C, R, Rem) native to the UI's compact table design.

## 📝 Appendix B: `reflection.md`
*Here are my notes for reflection.md*

> # Project Reflection
> 
> **1. How did your prompt evolve across iterations?**
> My prompt evolved from a basic data-extraction task into a complex, multimodal analytical engine within AI Studio. In Version 1, the JSON output was entirely generic. In Version 2, I introduced a constraint to cross-reference the incident context, forcing the model to act as a truth-checker to ensure top-level KPI fields reflected actual forensic realities rather than executive spin. In Version 3, I finalized the prompt by importing a screenshot of the remediation timeline from my Stitch UI. I explicitly instructed the model to analyze this visual asset and improve the data payload by calculating estimated timescales, sorting tasks by priority order, and creating a definitive key to summarize the UI's shorthand card labels (C, R, Rem).
> 
> **2. What model selection trade-offs did you observe?**
> I initially considered utilizing legacy models (like Gemini 1.5 Pro) to ensure deep reasoning accuracy for the conflict resolution tasks. However, I elected to use the **Gemini 3.1 Flash Preview**. The primary trade-off was working with a preview build—which required highly specific system instructions to guarantee strict JSON schema adherence—but this was vastly outweighed by the performance gains. The 3.1 Flash Preview demonstrated advanced reasoning capabilities, successfully correlating technical attack phases into priority queues, while maintaining the ultra-low Time-to-First-Token (TTFT) necessary for seamless dashboard rendering.
> 
> **3. How did Stitch accelerate or constrain UX design?**
> Stitch drastically accelerated the wireframing process. By pasting my structured JSON output into Stitch, I was able to instantly generate a high-fidelity dashboard featuring a high-contrast minimalist theme (light-gray backgrounds with stark black cards and neon green accents). However, this rigid UI introduced data constraints. The Stitch layout utilized extremely compact table rows for the timeline, which did not natively explain abbreviations or handle unprioritized lists well. This constrained my AI design, forcing me to shift the formatting burden onto the AI Studio prompt—compelling the model to intelligently generate a legend key and sort task priority before the data ever reached the UI.
> 
> **4. What would you change to productionise this system?**
> To move PIPA from a prototype to a production-ready system, I would implement Role-Based Access Control (RBAC) via Firebase Auth, leveraging the bottom navigation bar to restrict the "Risk" and "Timeline" views to users with specific clearance levels. I would also replace the manual PDF uploads in AI Studio with a live webhook integration into the company’s SIEM tool to dynamically populate the top-level incident KPIs. Finally, I would deploy the backend to Cloud Run within a secure VPC (Virtual Private Cloud) to ensure that highly sensitive, unredacted forensic data remains strictly isolated from public networks.
